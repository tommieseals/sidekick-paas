1.3 RAM-Aware Task Routing
This is a critical improvement from Grok based on real thrashing incidents. Every scheduled task and admin role must check available RAM before executing heavy workloads.

RAM Check Function (bash — for Mac Mini)
#!/bin/bash
# get_free_ram.sh - Returns free RAM in MB on macOS
get_free_ram() {
  local page_size=$(sysctl -n hw.pagesize)
  local free_pages=$(vm_stat | awk '/Pages free/ {gsub(/\./, "", $3); print $3}')
  local inactive_pages=$(vm_stat | awk '/Pages inactive/ {gsub(/\./, "", $3); print $3}')
  echo $(( (free_pages + inactive_pages) * page_size / 1048576 ))
}
 
free_ram=$(get_free_ram)
echo "Free RAM: ${free_ram}MB"
 
if [ "$free_ram" -lt 5000 ]; then
  echo "WARNING: Low RAM (${free_ram}MB). Offloading to Dell."
  # Route task to Dell via Tailscale Ollama endpoint
  export OLLAMA_HOST="<dell_tailscale_ip>:11434"
elif [ "$free_ram" -lt 3000 ]; then
  echo "CRITICAL: RAM ${free_ram}MB. Skipping heavy tasks."
  exit 1
fi

2.3 Shared Memory System
This is the persistence layer that survives resets. Every admin role writes structured entries to JSON files in ~/shared-memory/ on the Mac Mini hub. This directory is synced to Dell via Tailscale.

Directory Structure
~/shared-memory/
├── security.json     # Security Admin insights, alerts, scan results
├── network.json      # Network Admin status, latency logs, Tailscale health  
├── systems.json      # Systems Admin hardware status, RAM/CPU trends
├── dta.json           # DTA findings, tickets received, recommendations
├── tickets.json       # Cross-role tickets (queue for DTA processing)
└── backlog.json       # Unprocessed items surviving resets

Log Entry Format
{
  "timestamp": "2026-02-07T14:30:00Z",
  "source": "security_admin",
  "type": "TICKET",
  "target": "dta",
  "severity": "important",
  "summary": "New OpenClaw CVE published - evaluate patch",
  "details": "CVE-2026-XXXX affects versions < 2.3.1",
  "action_required": true,
  "status": "open"
}

🔄 Reset Recovery
When any admin role starts up after a crash or reset, its FIRST action must be: read its own shared-memory JSON file to rebuild context. This is the “memory” that survives reboots. The DTA should also read tickets.json and backlog.json to pick up any unprocessed work.

4.1 Video Transcription Pipeline (Fixed)
The original pipeline had no retry logic and could fail on flaky downloads. Improved version:

#!/bin/bash
# transcribe-video.sh - Download and transcribe YouTube videos for DTA analysis
# Improvements: retry logic, RAM check, chunking for long videos, fallback model
 
set -euo pipefail
 
VIDEO_URL="$1"
OUTPUT_DIR="${2:-~/dta-transcripts}"
mkdir -p "$OUTPUT_DIR"
 
# ── RAM Check ──
source ~/scripts/get_free_ram.sh
free_ram=$(get_free_ram)
if [ "$free_ram" -lt 3000 ]; then
  echo "ERROR: Insufficient RAM (${free_ram}MB). Deferring transcription."
  echo "{\"timestamp\": \"$(date -u +%FT%TZ)\", \"status\": \"deferred\", \"reason\": \"low_ram\", \"url\": \"$VIDEO_URL\"}" >> ~/shared-memory/backlog.json
  exit 1
fi
 
# ── Download Audio (with retries) ──
AUDIO_FILE="$OUTPUT_DIR/audio_$$.m4a"
echo "Downloading audio from $VIDEO_URL..."
yt-dlp \
  --retries 3 \
  --fragment-retries 3 \
  --extract-audio \
  --audio-format m4a \
  --audio-quality 5 \
  -o "$AUDIO_FILE" \
  "$VIDEO_URL" || {
    echo "ERROR: yt-dlp failed after retries. Logging to backlog."
    echo "{\"timestamp\": \"$(date -u +%FT%TZ)\", \"status\": \"failed\", \"reason\": \"download_failed\", \"url\": \"$VIDEO_URL\"}" >> ~/shared-memory/backlog.json
    exit 1
  }
 
# ── Transcribe (Whisper, with duration check) ──
DURATION=$(ffprobe -v error -show_entries format=duration -of csv=p=0 "$AUDIO_FILE" | cut -d. -f1)
TRANSCRIPT_FILE="$OUTPUT_DIR/transcript_$$.txt"
 
if [ "$DURATION" -gt 3600 ]; then
  echo "Long video (${DURATION}s). Chunking for transcription..."
  # Split into 30-minute chunks
  ffmpeg -i "$AUDIO_FILE" -f segment -segment_time 1800 -c copy "$OUTPUT_DIR/chunk_%03d.m4a"
  for chunk in "$OUTPUT_DIR"/chunk_*.m4a; do
    whisper "$chunk" --model base --output_format txt --output_dir "$OUTPUT_DIR" 2>/dev/null
    cat "$OUTPUT_DIR/$(basename $chunk .m4a).txt" >> "$TRANSCRIPT_FILE"
    rm "$chunk"
  done
else
  whisper "$AUDIO_FILE" --model base --output_format txt --output_dir "$OUTPUT_DIR" 2>/dev/null
  mv "$OUTPUT_DIR/$(basename $AUDIO_FILE .m4a).txt" "$TRANSCRIPT_FILE"
fi
 
rm -f "$AUDIO_FILE"
echo "Transcription complete: $TRANSCRIPT_FILE (${DURATION}s of audio)"
echo "$TRANSCRIPT_FILE"

4.2 DTA Summarizer (with Fallback Model)
#!/bin/bash
# summarize-for-dta.sh - Summarize a transcript using local Ollama, with model fallback
# Improvements: model fallback chain, RAM check, structured output
 
set -euo pipefail
 
TRANSCRIPT_FILE="$1"
SUMMARY_FILE="${TRANSCRIPT_FILE%.txt}.summary.json"
 
# ── RAM Check ──
source ~/scripts/get_free_ram.sh
free_ram=$(get_free_ram)
OLLAMA_TARGET="http://localhost:11434"
 
if [ "$free_ram" -lt 5000 ]; then
  echo "Low RAM on Hub. Routing to Dell worker..."
  OLLAMA_TARGET="http://<dell_tailscale_ip>:11434"
fi
 
# ── Model Fallback Chain ──
MODELS=("qwen2.5:7b" "phi3:mini" "gemma2:2b")
CONTENT=$(cat "$TRANSCRIPT_FILE" | head -c 15000)  # Limit context size
 
PROMPT="Summarize this video transcript for a Digital Transformation Administrator. \
Extract: 1) Key technologies mentioned, 2) Actionable recommendations for a home data center, \
3) Tools/services worth investigating, 4) Estimated relevance (High/Medium/Low). \
Respond in JSON format.\n\nTranscript:\n$CONTENT"
 
for model in "${MODELS[@]}"; do
  echo "Trying model: $model on $OLLAMA_TARGET"
  RESPONSE=$(curl -s --max-time 120 "$OLLAMA_TARGET/api/generate" \
    -d "{\"model\": \"$model\", \"prompt\": \"$PROMPT\", \"stream\": false}" \
    | jq -r '.response // empty' 2>/dev/null) || true
  
  if [ -n "$RESPONSE" ] && [ "$RESPONSE" != "null" ]; then
    echo "$RESPONSE" > "$SUMMARY_FILE"
    echo "Summary generated with $model: $SUMMARY_FILE"
    exit 0
  fi
  echo "Model $model failed or empty response. Trying next..."
done
 
echo "ERROR: All models failed. Logging to backlog."
echo "{\"timestamp\": \"$(date -u +%FT%TZ)\", \"status\": \"summarize_failed\", \"file\": \"$TRANSCRIPT_FILE\"}" >> ~/shared-memory/backlog.json
exit 1

4.3 Inbox Processor (RAM-Aware)
Add this guard to the top of any inbox/email batch processing script:

# ── Inbox Processor Guard ──
source ~/scripts/get_free_ram.sh

    >> ~/shared-memory/systems.json
  exit 0  # Exit cleanly, will retry next cycle
fi
 
# Also limit batch size to prevent RAM spike

launchctl load ~/Library/LaunchAgents/com.tommie.dta.weekly-scan.plist

5.2 Low-RAM Skip Pattern
Every launchd script should source the RAM check at the top. If RAM is insufficient, the script exits cleanly (exit 0) so launchd doesn’t mark it as failed, and it will simply retry at the next scheduled interval.
